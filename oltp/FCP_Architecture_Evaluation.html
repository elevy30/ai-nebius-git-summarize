FCP Architecture Evaluation

Can the Financial Crime Platform Run End-to-End on Databricks?
GoTyme Bank / GoTymeX · Architecture Review · March 2026
Flow Diagrams & Latency Databricks Fit Review

What Works Well

The shift-left principle is sound. Keeping logic at the data layer eliminates the #1 problem in FinCrime platforms: data movement creating inconsistency and latency. The medallion architecture (Bronze→Silver→Gold) is a proven pattern. Unity Catalog as the single governance plane is the right call for dual-jurisdiction (PH/SA).

Critical Gaps & Risks

8 architectural risks identified, ranked by severity and impact on production readiness.

#1 The Sub-50ms Fraud Path — Highest Risk Item

The claim: Transaction → API Gateway → Model Serving Endpoint → Lakebase Silver (feature lookup) → Brain → Response in <50ms.

The reality: This chain has at minimum 4 network hops:

  • API Gateway → MSE: ~5-10ms (network + cold routing)
  • MSE → Lakebase Silver (feature lookup via SQL): This is the bottleneck. Realistic P50 is 3-8ms, but P99 under load could be 20-40ms
  • MSE inference: ~2-5ms (provisioned)
  • MSE → Brain → response: ~5-10ms
Total realistic P99: 40-80ms, not sub-50ms. The feature lookup from Lakebase is not sub-10ms at P99 — especially under concurrent fraud scoring load.
Question for the team: Has anyone benchmarked Lakebase read latency at P99 under concurrent load? Redis/Memcached as a feature cache would be more realistic for sub-50ms — but that breaks the "no data movement" principle.

#2 Lakebase as Both OLTP and Feature Store — Architecture Tension

The doc uses Lakebase for: Bronze raw ingest (write-heavy), Silver operational tables (read/write), feature store for real-time fraud scoring (read-heavy, latency-critical), Flowable's backend database, and PuppyGraph source.

Risk: These are fundamentally different workload profiles. A burst of TM batch writes to Silver can degrade the fraud feature lookup P99 at the worst possible time.
Recommendation: Separate Lakebase projects for: Hot path (fraud features, provisioned), Operational (Flowable, case mgmt), Ingest (Bronze writes).

#3 "The Brain" — Insufficiently Defined

The Brain is described as the single decisioning core, but the doc doesn't specify:

  • What is it actually? A Databricks notebook? A long-running Spark job? A Python service? A stored procedure?
  • How does it receive real-time triggers? Databricks doesn't have a native event-driven invocation mechanism for notebooks.
  • How does it write back to Mambu/PSP/INC? These are synchronous write-backs to external core banking systems.
This is the biggest architectural ambiguity in the entire deck.

#4 Flowable ↔ Lakebase Integration — Untested at Scale

Flowable is traditionally deployed on standard PostgreSQL. Lakebase is Neon-based with ephemeral compute (cache lost on restart), higher write latency (WAL goes to safekeepers), and no support for some Postgres features (logical replication).

Flowable's internal schema is write-heavy. Flowable at scale on Neon-based Postgres is completely unproven territory. A single slow WAL flush could stall a BPMN process execution.

#5 PuppyGraph Zero-Copy — Needs Validation

"Zero-copy" on Lakebase means PuppyGraph issues SQL queries — it's not reading memory pages directly. Graph traversals (2nd/3rd degree) generate many sequential queries. At 25M customers with N-year history, the graph could have billions of edges.

Question: What is PuppyGraph's actual query pattern against Lakebase? Is it caching graph topology in its own memory, or is every traversal hitting Postgres?

#6 Zingg.ai Entity Resolution — Batch-Only

Onboarding is real-time — a new customer needs a party_id before CDD can start. If Zingg.ai is batch (hourly/daily), there's a window where a new customer exists without deduplication.

During that window, sanctions screening runs against an unresolved entity — potentially a false negative (regulatory violation).

#7 Dual Jurisdiction Data Residency — Not Addressed

  • Where does data physically reside? One Databricks workspace or two?
  • Can PH customer data be processed in SA compute, or is there a data residency constraint?
  • Unity Catalog ABAC is "in Preview" — is Preview acceptable for a regulated FinCrime platform?

#8 Gold Layer — Delta Lake vs Lakebase Confusion

Bronze/Silver are PostgreSQL in Lakebase, Gold is Delta Lake. But how does data move from Silver to Gold? That's ETL — the very thing shift-left is supposed to eliminate.

Questions: Who runs this ETL? On what schedule? During the ETL window, Gold is stale — meaning regulatory reporting views and the ML feature store are behind.

Questions for the Product Team / CTO

18 clarification questions organized by category. These should be answered before committing to the architecture.

Architecture Fundamentals

1
"What exactly IS the Brain?" — Is it a long-running service, a Spark job, a notebook triggered by events, or a set of Lakebase stored procedures? How does it receive real-time triggers from MSE and respond synchronously?
2
"How does the real-time fraud path work synchronously end-to-end?" — Draw the sequence diagram with actual millisecond budgets per hop. Where does the HTTP request enter, and where does the approve/deny response exit?
3
"What happens when Lakebase compute restarts?" — pg_prewarm reloads the cache, but during the reload window fraud scoring hits cold storage. What's the degradation plan?
4
"How does data move from Silver to Gold?" — If it's an ETL job, what's the freshness guarantee? This directly affects regulatory reporting accuracy.

Latency & Scale

5
"Has anyone benchmarked Lakebase read latency at P99 under concurrent write load?" — Not P50, not in isolation. P99 while Bronze ingest, Flowable state writes, and fraud feature reads are all happening.
6
"What is the target TPS for the fraud path?" — 25M customers, but how many transactions per second? 100 TPS is very different from 10,000 TPS.
7
"What is the failover story?" — If Lakebase compute goes down, is there automatic failover? What's the RTO? Even 30 seconds of downtime means 30 seconds of unscored transactions.

Component Choices

8
"Why Flowable and not Databricks Workflows?" — Flowable adds significant complexity. What does Flowable provide that Databricks Workflows + Delta Live Tables cannot?
9
"Why PuppyGraph and not GraphFrames (native Spark)?" — GraphFrames runs natively on Spark. Where specifically does PuppyGraph's real-time traversal justify the additional component?
10
"Why Superblocks for case UI?" — If Flowable Work already provides a case management UI, why add Superblocks? Two UX frameworks = two things to maintain.
11
"What is Zingg.ai's real-time dedup story for onboarding?" — If batch, how long until a resolved party_id? Is there an interim deterministic match?

Regulatory & Governance

12
"Is Unity Catalog ABAC (Preview) acceptable for BSP/SARB?" — Preview features can change. What's the fallback?
13
"Who owns the 7-year audit trail retention policy?" — If someone runs VACUUM with a short retention, the audit trail is gone. What prevents this?
14
"How do you prove a rule change didn't retroactively affect past decisions?" — Is DMN versioning + Delta time-travel actually tested, or theoretical?

Missing from the Deck

15
"Where is the disaster recovery plan?" — The entire platform is on Databricks. What if the region goes down?
16
"Where is the data migration plan?" — The deck describes target state but not the path from current state.
17
"What are the cost estimates?" — Provisioned Lakebase 24/7, autoscaling batch, MLflow MSE, PuppyGraph, Flowable infra. Has anyone modeled the monthly bill?
18
"What is the team structure?" — Hafnia needed 10 people for a simpler use case. How many engineers does GoTyme need for FCP?

Bottom Line Assessment

The architecture is ambitious and conceptually sound but has three critical unknowns that must be validated before committing:

  1. The real-time fraud path latency — needs actual benchmarking, not theoretical estimates
  2. The Brain's runtime model — this is architecturally undefined and is the central decision component
  3. Flowable on Lakebase at scale — completely unproven combination
Recommendation: Focused PoC on the fraud path (items 1+2) before investing in the full platform build. If sub-50ms can't be met with Lakebase alone, the architecture needs a caching layer — which changes the entire shift-left premise.
