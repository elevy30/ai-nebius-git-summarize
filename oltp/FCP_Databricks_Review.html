Databricks Fit Review

Where Databricks is the right tool, where it's stretched, and where it's wrong
GoTyme Bank / GoTymeX · FCP Architecture · March 2026
Gaps & Risks Flow Diagrams & Latency

The Core Tension

The FCP deck makes a bold claim: everything runs on Databricks. That's the shift-left pitch. But Databricks is fundamentally an analytics and ML platform that recently added OLTP capabilities (Lakebase). The FCP is asking it to also be a real-time transaction processing system, a workflow engine backend, and an application database — simultaneously.

Where Databricks is GENUINELY the Best Choice

1. The Gold Layer — Analytics, Regulatory Reporting, Audit

Confidence: 10/10

This is Databricks' home turf. Delta Lake with time-travel for 7-year regulatory archives, Z-order for fast BSP/SARB examination queries, Unity Catalog for governance — there is no better platform.

  • STR/CTR canonical views: SQL on Delta Lake Gold — perfect
  • goAML reporting dataset: Spark batch producing regulatory XML — perfect
  • MIS / Board-level dashboards: AI/BI Dashboards on Gold — perfect
  • Rule replay against historical data: Delta time-travel + notebooks — perfect

2. ML Model Training and Batch Scoring

Confidence: 10/10
  • Fraud model training on historical transaction data — MLflow + Spark, excellent
  • CRR batch scoring — Spark notebook, excellent
  • TM typology feature engineering — Spark batch, excellent
  • Champion/challenger model comparison — MLflow experiment tracking, excellent

3. AML Transaction Monitoring (Batch)

Confidence: 9/10

Overnight TM batch is a perfect Databricks workload: Spark SQL + notebooks on a schedule, Databricks Jobs for orchestration. Only risk is complexity if there are 50+ interdependent jobs.

4. Entity Resolution (Zingg.ai)

Confidence: 8/10 batch · 0/10 real-time

Zingg.ai runs natively on Spark. Databricks is the right platform. The issue isn't Databricks — it's that Zingg is batch-only while onboarding needs real-time dedup.

5. Data Ingestion (Bronze)

Confidence: 8/10

Lakeflow Connect for ingesting from core banking (Mambu), LSEG Refinitiv, Darwinium, PSP. Depends on connector maturity for specific FinCrime sources.

Where Databricks is Being STRETCHED

6. Lakebase as the Feature Store for Real-Time Fraud

Confidence: 5/10 (current) · 8/10 (with Online Feature Store)

Using Lakebase Silver as a low-latency feature store means fighting its serverless design: pg_prewarm + provisioned compute + disabled scale-to-zero.

Recommendation: Use Databricks Online Feature Store (online tables backed by DynamoDB/Cosmos) for the fraud hot path. Features computed in Databricks, governed by Unity Catalog, served from an optimized read path with guaranteed sub-5ms reads.

7. Model Serving Endpoints for Real-Time Fraud

Confidence: 6/10

MSE for inference is fine. The problem is: who holds the HTTP connection?

The flow Mobile → API Gateway → ??? → MSE → Lakebase → Brain → ??? needs a lightweight orchestration microservice (Python/Go) that Databricks doesn't provide. The deck doesn't acknowledge this gap.

8. "The Brain" on Databricks

Confidence: 3/10

This is the weakest part of the architecture. The Brain on "Databricks provisioned compute" has no viable runtime:

  • Notebook? — Not for real-time HTTP request/response
  • Long-running Spark job? — ~50ms overhead per task scheduling alone
  • Python service on a cluster? — Anti-pattern; paying for Spark to run Flask
Recommendation: The Brain should be a lightweight stateless FastAPI service (on K8s or Databricks Apps) that receives decisions from MSE/Flowable, applies threshold rules, writes audit to Lakebase, and calls external systems. This is a microservice — and that's OK.
Where Databricks is the WRONG Tool

9. Flowable's Backend Database

Confidence: 2/10

Flowable writes heavily — every BPMN state transition, task assignment, and timer event. Lakebase (Neon-based) has:

  • Higher write latency (WAL goes to remote safekeepers)
  • Ephemeral compute (cache lost on restart)
  • No logical replication support
Recommendation: Run Flowable on standard managed Postgres (RDS/Cloud SQL). Flowable reads FCP data FROM Lakebase via JDBC, but its own internal state lives on a standard Postgres.

10. Real-Time Event Routing / Message Bus

Confidence: 2/10

The architecture has no message bus, but needs async signals for:

  • Brain → external systems (Mambu, PSP, INC) write-back
  • Screening list update → re-screening trigger
  • Batch job completion → Flowable case creation
Recommendation: Add Kafka or EventBridge for event routing. Databricks "Zerobus" is too early for production FinCrime workloads.

Recommended Architecture Adjustment

KEEP ON DATABRICKS (its strengths)

  • Bronze → Silver → Gold medallion (Lakebase + Delta Lake)
  • All ML training + batch scoring (MLflow + Spark)
  • AML Transaction Monitoring batch (Spark Jobs)
  • Entity Resolution (Zingg.ai on Spark)
  • CRR scoring batch (Spark notebooks)
  • Feature engineering + feature store (Online Feature Store)
  • Gold regulatory reporting + audit (Delta Lake)
  • Unity Catalog governance (everything)
  • AI/BI dashboards

MOVE OFF DATABRICKS (not its strengths)

  • Flowable backend DB → Standard managed Postgres
  • The Brain → Lightweight Python service (K8s or Databricks Apps)
  • Fraud orchestration → Thin API service calling MSE + Feature Store
  • Event routing → Kafka or EventBridge
  • Real-time feature serving → Databricks Online Tables (not raw Lakebase SQL)

The Bottom Line

Databricks should be the data platform, ML platform, and governance platform — that's 70-80% of the FCP. But it should NOT be the application runtime for real-time orchestration, workflow engine hosting, or event routing.

The shift-left principle still holds: all data, all ML, all rules-as-data, all governance live in Databricks. But the thin orchestration layer (The Brain, event routing, Flowable runtime) should run on purpose-built infrastructure that reads from and writes to Databricks.

This isn't a compromise — it's the correct architecture. Even Databricks' own reference architectures for real-time applications show a lightweight serving layer in front of the platform.
